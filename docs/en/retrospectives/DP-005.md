# DP-005 Runtime Message Router Retrospective

[Russian version](../../ru/retrospectives/DP-005.md)

**Status:** Completed

## Objective

DP-005 introduced deterministic configuration-driven selection of exactly one Runtime Message Handler or an explicit No Match outcome. The completed subsystem connects declarative Routing metadata, an immutable Runtime Snapshot, startup-time Router compilation, Session-created Runtime Message Context, and synchronous Handler execution without exposing WebSocket transport to Router.

## What Worked Well

- Dividing the work into small contracts kept every stage buildable and independently reviewable.
- Immutable Configuration, Context, Snapshot, and compiled Router values established clear ownership boundaries.
- Separate validation at the Control Plane, Snapshot publication, and Router compilation boundaries prevented malformed programmatic input from entering Runtime.
- Unit, concurrency, allocation, startup-failure, and real WebSocket integration tests provided complementary proof of the subsystem invariants.
- Stopping implementation when the absent-Routing construction contract was ambiguous prevented an implicit architecture decision in production code.

## Architecture Decisions That Proved Correct

- Implementing Router as `message.Handler` preserved the existing synchronous Session execution and error-propagation path.
- Compiling, resolving, and sorting Routes once before Listener startup kept the message hot path read-only and allocation-free at the Router boundary.
- Distinguishing absent Routing from present-empty Routing preserved both backward compatibility and explicit reject-all configuration.
- Keeping `router.New` strict while providing a separate compatibility factory maintained one meaning for declarative compilation.
- Restricting the initial Handler registry to the finite `legacy` binding made startup executability explicit and prevented dynamic registry lifecycle concerns.
- Defining No Match as a normal nil result avoided synthetic discard Handlers and preserved the Session read loop.

## Corrections Made During Implementation

- The compiled Route representation was corrected to store enabled Routes in ascending Priority rather than declaration order. This made the compiled state directly express the approved selection rule.
- The absent-Routing path was separated from strict Router compilation. A dedicated compatibility factory now represents zero Routes and an optional legacy Default Handler without weakening `router.New` validation.
- Runtime integration was completed through one startup construction branch and one post-startup `Router.Handle` path, eliminating the possibility of bypassing Router for compatibility behavior.

## Lessons Learned

- Configuration absence, explicit emptiness, invalid input, and No Match are separate states and require separate normative semantics.
- Compile-once architecture must be visible in the stored representation, not only implied by selection code.
- Immutability requires detached ownership at every publication boundary, including nested collections and accessor results.
- Hot-path requirements need direct allocation, synchronization, and repeated-dispatch proof in addition to functional tests.
- Compatibility behavior should be represented by an explicit construction contract rather than hidden fallback during execution.
- Cross-layer validation is strongest when each layer proves its assigned responsibility and end-to-end tests prove their composition.

## Recommendations for Future Design Proposals

- Define configuration presence semantics and compatibility behavior before implementation begins.
- Specify the exact compiled representation, ordering rules, and construction boundary when operational performance depends on precomputation.
- Assign validation, normalization, reference resolution, and execution to explicit component boundaries.
- Define nil, default, no-match, and error outcomes together with their downstream lifecycle effects.
- Include package-boundary, immutability, concurrency, and allocation evidence in the initial test strategy.
- Require an end-to-end proof path from published Configuration through Runtime execution for every configuration-driven subsystem.
