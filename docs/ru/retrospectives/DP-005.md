# Ретроспектива DP-005 Runtime Message Router

[English version](../../en/retrospectives/DP-005.md)

**Статус:** Completed

## Цель

DP-005 добавил детерминированный выбор ровно одного Runtime Message Handler либо явного результата No Match на основе Configuration. Завершённая подсистема связывает декларативные Routing metadata, immutable Runtime Snapshot, компиляцию Router во время startup, создаваемый Session Runtime Message Context и синхронное выполнение Handler без раскрытия WebSocket transport компоненту Router.

## Что сработало хорошо

- Разделение работы на небольшие контракты сохраняло каждый этап компилируемым и независимо проверяемым.
- Immutable значения Configuration, Context, Snapshot и compiled Router задали явные границы ownership.
- Раздельная валидация на границах Control Plane, публикации Snapshot и компиляции Router не позволила некорректному programmatic input попасть в Runtime.
- Unit-, concurrency-, allocation-, startup-failure- и реальные WebSocket integration tests дали взаимодополняющие доказательства инвариантов подсистемы.
- Остановка реализации при неоднозначном контракте отсутствующей Routing-секции предотвратила появление неявного архитектурного решения в production-коде.

## Архитектурные решения, подтвердившие корректность

- Реализация Router в роли `message.Handler` сохранила существующий синхронный Session execution path и передачу ошибок.
- Однократные компиляция, resolution и сортировка Routes до Listener startup сохранили message hot path read-only и allocation-free на границе Router.
- Различие между отсутствующей и явно пустой Routing-секцией сохранило обратную совместимость и явную reject-all Configuration.
- Сохранение строгого `router.New` вместе с отдельной compatibility factory закрепило единственное значение декларативной компиляции.
- Ограничение initial Handler registry конечной привязкой `legacy` сделало startup executability явной и исключило lifecycle-вопросы динамического registry.
- Определение No Match как штатного nil-результата позволило не создавать synthetic discard Handler и сохранило Session read loop.

## Корректировки во время реализации

- Compiled Route representation был скорректирован так, чтобы хранить enabled Routes по возрастанию Priority, а не в declaration order. Благодаря этому compiled state непосредственно выражает утверждённое правило выбора.
- Путь отсутствующей Routing-секции был отделён от строгой компиляции Router. Отдельная compatibility factory теперь представляет ноль Routes и optional legacy Default Handler, не ослабляя валидацию `router.New`.
- Runtime integration была завершена через одну startup construction branch и единый post-startup путь `Router.Handle`, что исключило обход Router для compatibility-поведения.

## Извлечённые уроки

- Отсутствие Configuration, явная пустота, некорректный input и No Match являются разными состояниями и требуют отдельных нормативных semantics.
- Compile-once architecture должна быть видна в хранимом representation, а не только подразумеваться кодом выбора.
- Immutability требует detached ownership на каждой границе публикации, включая вложенные collections и результаты accessors.
- Требования к hot path нуждаются в прямых allocation-, synchronization- и repeated-dispatch proofs в дополнение к функциональным тестам.
- Compatibility-поведение следует представлять явным construction contract, а не скрытым fallback во время execution.
- Cross-layer validation наиболее убедительна, когда каждый слой доказывает назначенную ему ответственность, а end-to-end tests доказывают их композицию.

## Рекомендации для будущих Design Proposals

- Определять semantics присутствия Configuration и compatibility-поведение до начала реализации.
- Фиксировать точное compiled representation, правила ordering и construction boundary, если operational performance зависит от precomputation.
- Назначать validation, normalization, reference resolution и execution явным component boundaries.
- Определять nil-, default-, no-match- и error outcomes вместе с их влиянием на downstream lifecycle.
- Включать доказательства package boundaries, immutability, concurrency и allocations в первоначальную test strategy.
- Требовать end-to-end proof path от Published Configuration до Runtime execution для каждой configuration-driven подсистемы.
